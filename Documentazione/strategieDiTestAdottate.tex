La suite di test del progetto DietiEstates2025 adotta un approccio multi-livello combinando 
diverse strategie di testing secondo il modello Testing Pyramid:
\begin{enumerate}
    \item Unit Testing (base): Isolamento dei componenti singoli
    \item Integration Testing (middle): Test di flussi completi
    \item Controller Testing (top): Test dell'interfaccia HTTP
\end{enumerate}

Sono stati individuati diversi scenari positivi e negativi
\begin{enumerate}
    \item Scenari positivi verificano che le operazioni funzionino correttamente con input validi, 
    coprendone il maggior numero per singolo test
    \item Scenari negativi testano la gestione degli errori, isolando ogni singolo input errato per singolo test   
\end{enumerate}

Ogni test segue rigorosamente il pattern Arrange, Act, Assert:
\begin{enumerate}
    \item Arrange: setup dei mock e dei dati di test
    \item Act: esecuzione del metodo da testare
    \item Assert: verifica dei risultati attesi
\end{enumerate}

Sono state testate le seguenti funzionalità
\begin{enumerate}
    \item Registrazione
    \item Ricerca immobili
    \item Caricamento immobili
    \item Avvio di una trattativa
\end{enumerate}

La maggior parte dei test prende ispirazione dai principi del WECT (Weak Equivalence Class 
Testing), mantenendo costanti gli altri valori di input per isolare il comportamento sotto 
esame. Tuttavia, i casi di test non si limitano a una semplice partizione formale del dominio
degli input, ma seguono anche una scomposizione logica dei possibili scenari di utilizzo del
sistema. In particolare, vengono considerati casi significativi dal punto di vista applicativo,
come ad esempio il tentativo di caricamento di un immobile duplicato, caricare un offerta con 
saldo negativo, inserire un immobile senza un immagine principale, che riflettono situazioni 
realistiche di errore o uso improprio.

In questo modo, la progettazione dei test combina una partizione del dominio degli input con 
una selezione mirata di scenari applicativi rilevanti, ottenendo una copertura più 
significativa del comportamento del sistema rispetto a un’applicazione puramente formale del 
WECT.
La suite utilizza @MockBean per isolare il service layer dalle sue dipendenze 
(repository, MinIO service). Questo permette di testare la logica del servizio senza dipendere 
da database o servizi esterni reali.
